# Project Requirements

เอกสารนี้สรุป requirement ของระบบจากโค้ดและโครงสร้างโปรเจกต์ปัจจุบัน เพื่อใช้เป็น baseline สำหรับการพัฒนาต่อ

## 1. เป้าหมายของระบบ

ระบบนี้ต้องรองรับการขายสินค้าดิจิทัลสำหรับตลาดไทย เช่น ไอดีเกม, รหัสเกม, หรือสินค้าดิจิทัลอื่น ๆ ผ่านเว็บไซต์ โดยมีทั้งฝั่งลูกค้าและฝั่งผู้ดูแลระบบ

ระบบต้องช่วยให้:

- ลูกค้าค้นหาสินค้า ดูรายละเอียดสินค้า ซื้อสินค้า และติดตามข้อมูลของตนเองได้
- ลูกค้าสมัครสมาชิก, เข้าสู่ระบบ, เติมเงิน, ใช้เครดิต และใช้งานฟีเจอร์สะสมแต้มได้
- ผู้ดูแลระบบจัดการสินค้า, ผู้ใช้, ข่าวสาร, การเติมเงิน, กาชา, สิทธิ์การเข้าถึง และการตั้งค่าเว็บไซต์ได้
- ระบบมีมาตรการด้านความปลอดภัยและตรวจสอบย้อนหลังได้

## 2. ประเภทผู้ใช้งาน

- ผู้เยี่ยมชม: ดูหน้าเว็บสาธารณะ, รายการสินค้า, ข่าวสาร, คู่มือ, หน้าเข้าสู่ระบบและสมัครสมาชิก
- สมาชิก: ซื้อสินค้า, เติมเงิน, ดูประวัติ, จัดการโปรไฟล์, ใช้งาน dashboard และฟีเจอร์กาชา
- ผู้ดูแลระบบ: จัดการข้อมูลระบบผ่าน admin console ตามสิทธิ์ที่ได้รับ

## 3. Functional Requirements ฝั่งลูกค้า

### 3.1 การสมัครสมาชิกและเข้าสู่ระบบ

- ผู้ใช้ต้องสามารถสมัครสมาชิกผ่านหน้า register ได้
- ผู้ใช้ต้องสามารถเข้าสู่ระบบและออกจากระบบได้
- ระบบต้องมีการจัดการ session สำหรับผู้ใช้ที่ล็อกอินแล้ว
- ระบบต้องรองรับการรีเซ็ตรหัสผ่าน

### 3.2 การดูสินค้าและเลือกซื้อ

- ผู้ใช้ต้องสามารถดูรายการสินค้าทั้งหมดจากหน้าร้านได้
- ผู้ใช้ต้องสามารถดูรายละเอียดสินค้าแต่ละรายการได้
- ระบบควรรองรับสินค้าแนะนำและสินค้าลดราคา
- ระบบต้องรองรับการซื้อสินค้าดิจิทัลผ่านกระบวนการสั่งซื้อ
- ระบบต้องจัดเก็บข้อมูลคำสั่งซื้อและข้อมูลที่มอบให้ลูกค้าหลังซื้อ

### 3.3 ตะกร้าสินค้าและโปรโมชั่น

- ผู้ใช้ต้องสามารถเพิ่มสินค้าในตะกร้าและทำรายการ checkout ได้
- ระบบต้องรองรับ promo code สำหรับส่วนลด
- ระบบต้องตรวจสอบเงื่อนไขการใช้ promo code เช่น วันหมดอายุและจำนวนครั้งที่ใช้ได้

### 3.4 เครดิต, แต้ม, และการเติมเงิน

- ระบบต้องมียอดเครดิตสำหรับการซื้อสินค้า
- ระบบต้องมี point balance สำหรับฟีเจอร์ที่ใช้แต้ม
- ผู้ใช้ต้องสามารถส่งสลิปเพื่อเติมเงินเข้าระบบได้
- ระบบต้องเก็บสถานะรายการเติมเงิน เช่น รอตรวจสอบ, อนุมัติ, ปฏิเสธ

### 3.5 โปรไฟล์และ dashboard ผู้ใช้

- ผู้ใช้ต้องสามารถดูข้อมูลบัญชีของตนเองได้
- ผู้ใช้ต้องสามารถแก้ไขข้อมูลโปรไฟล์บางส่วนได้
- ผู้ใช้ต้องสามารถดูประวัติคำสั่งซื้อ, ประวัติเติมเงิน, และข้อมูลที่เกี่ยวข้องกับบัญชีได้
- ระบบต้องมี dashboard สำหรับผู้ใช้ที่ล็อกอินแล้ว

### 3.6 ระบบระดับสมาชิก

- ระบบต้องเก็บยอดใช้งานสะสมของผู้ใช้
- ระบบต้องรองรับการคำนวณระดับสมาชิกตามเงื่อนไขของระบบ
- ระบบอาจแสดง badge หรือสถานะพิเศษของผู้ใช้ได้

### 3.7 ข่าวสารและคอนเทนต์หน้าเว็บ

- ระบบต้องแสดงข่าวสารหรือบทความประชาสัมพันธ์ได้
- ระบบต้องมีหน้าช่วยเหลือหรือ FAQ
- ระบบต้องแสดง popup, เมนูนำทาง, footer widget และคอนเทนต์หน้าแรกตามการตั้งค่าจากระบบหลังบ้าน

## 4. Functional Requirements ฝั่งกาชาและระบบพิเศษ

### 4.1 ระบบกาชา

- ระบบต้องรองรับตู้กาชาหลายตู้
- แต่ละตู้ต้องกำหนดต้นทุนการเล่นได้ เช่น ฟรี, เครดิต, หรือแต้ม
- แต่ละตู้ต้องกำหนดรางวัลและอัตราความน่าจะเป็นได้
- ระบบต้องเก็บประวัติการหมุนของผู้ใช้
- ระบบควรแสดงผลการหมุนล่าสุดหรือกิจกรรมล่าสุดของกาชาได้

### 4.2 Gacha Grid / SPIN_X

- ระบบต้องรองรับโหมดกาชาแบบ grid game
- ผลลัพธ์ต้องคำนวณจาก selector หรือ path intersection ตาม logic ของระบบ
- ระบบต้องสามารถกำหนดระดับรางวัล เช่น common, rare, epic, legendary

### 4.3 Season Pass และฟีเจอร์พิเศษ

- ระบบต้องรองรับ season pass และการรับ reward ที่เกี่ยวข้อง
- ระบบควรรองรับฟีเจอร์ส่งเสริมการมีส่วนร่วม เช่น referral, reward, หรือกิจกรรมเฉพาะระบบ

## 5. Functional Requirements ฝั่งผู้ดูแลระบบ

### 5.1 การจัดการสินค้า

- ผู้ดูแลระบบต้องสามารถสร้าง, แก้ไข, ดูรายการ, ทำซ้ำ, และจัดการสถานะสินค้าผ่านหน้า admin ได้
- ระบบต้องรองรับการจัดการ stock หรือข้อมูลลับของสินค้า
- ระบบต้องป้องกันข้อมูล stock ซ้ำในกรณีที่ระบบกำหนดไว้
- ระบบต้องรองรับการตั้งค่าให้สินค้าถูกลบอัตโนมัติหลังขายได้

### 5.2 การจัดการผู้ใช้และสิทธิ์

- ผู้ดูแลระบบต้องสามารถดูและจัดการข้อมูลผู้ใช้ได้
- ระบบต้องรองรับ role และ permissions สำหรับการเข้าถึงเมนูและ API ฝั่ง admin
- การมองเห็นเมนู admin อย่างเดียวไม่ถือว่าเพียงพอ ต้องมีการบังคับสิทธิ์ใน backend ด้วย

### 5.3 การจัดการรายการเติมเงิน

- ผู้ดูแลระบบต้องสามารถตรวจสอบและอนุมัติหรือปฏิเสธสลิปเติมเงินได้
- ระบบต้องเก็บข้อมูลการอ้างอิงและสถานะของรายการเติมเงิน

### 5.4 การจัดการคอนเทนต์และการตั้งค่าเว็บ

- ผู้ดูแลระบบต้องสามารถจัดการข่าวสาร, help article, popup, เมนูนำทาง, footer links, และการตั้งค่าหน้าเว็บได้
- ระบบต้องรองรับการตั้งค่า currency หรือสัญลักษณ์เงินที่ใช้แสดงผล

### 5.5 การจัดการกาชา

- ผู้ดูแลระบบต้องสามารถจัดการหมวดหมู่กาชา, ตู้กาชา, รางวัล, และค่าตั้งต้นของระบบกาชาได้
- ผู้ดูแลระบบต้องสามารถตั้งค่าเงื่อนไขการเล่น เช่น daily spin limit และ cost type ได้

### 5.6 การตรวจสอบและรายงาน

- ระบบควรมี dashboard หรือหน้าสรุปข้อมูลสำคัญสำหรับผู้ดูแลระบบ
- ระบบควรมี audit log สำหรับตรวจสอบกิจกรรมสำคัญ
- ระบบควรรองรับการ export ข้อมูลในบางส่วนที่จำเป็นต่อการดำเนินงาน

## 6. Security Requirements

- ระบบต้องมีการยืนยันตัวตนก่อนเข้าถึงข้อมูลหรือฟังก์ชันที่ต้องใช้สิทธิ์
- ระบบต้องบังคับ authorization สำหรับหน้า admin และ admin API ตาม role/permission
- ระบบต้องใช้ CSRF protection สำหรับคำขอที่มีผลต่อข้อมูล
- ระบบต้องมี validation ของข้อมูล input ทั้งฝั่ง client และ server ตามความเหมาะสม
- ระบบต้องจัดเก็บข้อมูลสำคัญอย่างระมัดระวัง โดยเฉพาะข้อมูลลับของสินค้า, สลิป, และข้อมูลผู้ใช้
- ระบบควรรองรับ rate limiting สำหรับ endpoint ที่มีความเสี่ยง
- ระบบต้องมี audit trail สำหรับกิจกรรมสำคัญบางประเภท

## 7. Non-Functional Requirements

- ระบบต้องพัฒนาอยู่บน Next.js, TypeScript, MySQL และ Drizzle ORM
- โค้ดควรแยกความรับผิดชอบระหว่าง route, component, validation, และ business logic ให้ดูแลง่าย
- ระบบต้องรองรับภาษาไทยและหลีกเลี่ยงปัญหา encoding
- UI ต้องใช้งานได้เหมาะกับเว็บอีคอมเมิร์ซสำหรับผู้ใช้ไทย
- ระบบควรมี test สำหรับ logic สำคัญตามความเหมาะสม
- ระบบควรพร้อมต่อยอดด้าน performance, observability, และ deployment quality checks

## 8. ข้อกำหนดเชิงข้อมูล

- ระบบต้องมีข้อมูลหลักอย่างน้อย ได้แก่ users, products, orders, topups, promo codes, site settings, news, nav items, footer links, gacha machines, และ gacha rewards
- ระบบต้องมีความสัมพันธ์ของข้อมูลที่รองรับทั้งฝั่งลูกค้าและฝั่ง admin
- ระบบต้องสามารถติดตามสถานะของธุรกรรมสำคัญ เช่น การซื้อและการเติมเงินได้

## 9. ข้อสมมติของเอกสารนี้

- เอกสารนี้อ้างอิงจาก implementation ที่มีอยู่ในโปรเจกต์ปัจจุบัน
- บาง feature อาจมีรายละเอียดเชิงธุรกิจมากกว่านี้ แต่ยังไม่ได้ระบุครบในเอกสารต้นทาง
- หากจะใช้เอกสารนี้เป็น PRD จริงสำหรับลูกค้าหรือทีม ควรแตกต่อเป็น user flow, acceptance criteria, และ business rules รายโมดูล

## 10. ข้อเสนอแนะสำหรับการใช้งานต่อ

- ใช้เอกสารนี้เป็น requirement baseline ก่อนเริ่มพัฒนาฟีเจอร์ใหม่
- เมื่อมีงานใหม่ ควรเพิ่มหัวข้อย่อยแบบนี้:
  - เป้าหมายของฟีเจอร์
  - ผู้ใช้งานที่เกี่ยวข้อง
  - เงื่อนไขการทำงาน
  - validation
  - permission
  - edge cases
  - acceptance criteria
